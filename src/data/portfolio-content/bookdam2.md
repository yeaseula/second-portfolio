## description

책담은 개인의 독서 기록을 지속 가능한 데이터로 쌓아가는 **가상 서재 서비스**입니다.

단순한 독서 메모를 넘어, 로그인 기반의 개인 공간에서 리뷰·위시리스트·독서 로그를 관리하며

독서 습관이 자연스럽게 이어지도록 설계했습니다.

Season 1에서는 Google 설문지와 스프레드시트를 활용해
최소한의 기록(Create)만 가능한 구조였으며,

데이터 수정·삭제와 개인화에는 명확한 한계가 있었습니다.
Season 2는 ‘개인화된 서비스’로 확장된 버전입니다.

회원 시스템 부재로 데이터가 남지 않던 구조를 개선하고,
개인별 데이터가 축적되는 환경을 중심으로 전체 구조를 다시 설계했습니다.

사용자는 가입 시 설정한 관심사 키워드를 기반으로 책을 추천받고, 위시리스트로 관리할 수 있습니다.

독서 후 리뷰를 남기면 캘린더에 기록이 남고,스톱워치를 통해 독서 시간을 측정하며
읽고, 남기고, 이어지는 독서 흐름을 경험하게 됩니다.

## skills


### 개발 환경 (Frontend Framework)

- **React**
  컴포넌트 단위 렌더링으로 인터랙티브한 독서 기록 서비스에 적합.

- **Next.js**
  서버 사이드 렌더링(SSR)으로 초기 렌더링 속도 개선.

- **TypeScript**
  데이터 타입과 상태를 명확히 구분, CRUD 구현 시 안정성 확보.

### 백엔드
- **Supabase**
  회원가입/로그인과 CRUD를 한 번에 처리 가능.
  SQL 기반 테이블로 데이터 관계 설계 용이.


### 상태관리
- **Zustand**
  간결한 API, TypeScript 지원, 커뮤니티 풍부.
  전역 상태 관리와 유지보수에 적합.

### 선택한 라이브러리
- **React Error Boundary**
  개인화 서비스 구조에서 에러 처리 복잡도를 최소화.

- **Swiper**
  접근성과 커뮤니티가 검증된 슬라이드 라이브러리.

  직접 접근성 구현 대비 유지보수 효율적.

- **React-hook-form**
  폼 유효성 검사와 관리가 간편하여 UX 향상.

**선택하지 않은 라이브러리**
- Lottie
네비게이션 영역에서 과도한 리소스라고 판단해 제거 후 CSS 인터렉션으로 대체.

## trouble

(문제)
초기 유효성 검사 - onBlur 처리
문제점 - UX 저하, 폼 신뢰성 하락

(1차 시도 / 실패)
onChange 방식으로 전환 - 리렌더 폭증

(2차 시도 / 실패)
state를 ref로 변경
useCallback, memo 적용
리렌더는 줄었지만 상태와 유효성 로직의 복잡성 증가로 유지보수의 난이도 상승

(전환점)
"리렌더가 항상 나쁜 것은 아니다" 라는 점을 인식.
중요한 건
"무거운 컴포넌트", "무거운(복잡한)연산의 재실행"

(해결)
react-hook-form 도입
- 필요한 필드만 리렌더
- 유효성 검사 로직을 대폭 축소
- 타입스크립트 지원으로 안전한 폼 생성 가능한

(결과)
- UX개선과 코드의 복잡도 감소
- input, textarea, select 등 폼에 필요한 필드의 분리로 협업 시에도 안전하고 빠른 개발 가능
