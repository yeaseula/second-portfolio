## trouble

> Tailwind 디자인 토큰 충돌 문제 해결

**문제**

- Tailwind cn() 유틸을 사용하는 환경에서 Typography 토큰과 기존 Tailwind class가 병합되어
  오버라이드 되지 않는 문제 발생

**원인 분석**

- Tailwind의 class merge 로직에서 text-\* 계열을 동일 속성으로 판단하여 마지막 클래스만 적용되는 구조를 확인

**해결 과정**

- 초기에는 커스텀 cn함수를 만들고 기존 디자인토큰 체계를 유지하고자 했으나
  논의를 통해 Figma 디자인 시스템의 명칭은 그대로 유지하는 것이 좋다고 결정
- Typography 토큰 네이밍 구조를 재정의 (기존 text-* 에서 typo-*로 재정의)

**결과**

- 디자인/개발 간 토큰 구조를 일치시켜 혼란 최소화
- 스타일 충돌 제거

> 확장 가능한 Modal 구조 설계 방식 개선

**문제**

- Modal 내부에 Form, Validation, Action 영역 등 다양한 구조 확장이 필요
- 직접 구현 시 접근성 및 상태 관리 복잡도 증가 예상

**해결 과정**

- 팀 회의를 통해 Custom 구현 대신 검증된 Radix UI 기반 라이브러리 활용 결정

**결과**

- Modal 확장 구조 확보
- 접근성 개선 및 유지보수 비용 감소

> Storybook 도입 초기 적응 및 컴포넌트 문서화

**문제**

- Storybook 사용 경험 부재로 초기 설정 및 스토리 작성 난이도 존재

**해결 과정**

- 기존 팀원의 스토리 구조 분석
- 공통 컴포넌트에 스토리 적용 후 컴포넌트 상태별 UI 확인 환경 구축

**결과**

- 컴포넌트 단위 개발 및 테스트 가능

> CI / Husky 환경 빌드 실패 해결

**문제**

- Pull Request 생성 후 CI 파이프라인에서 빌드 실패 발생

**원인 분석**

- 프로젝트에 적용된 Husky 및 CI 검증 과정에 필요한 의존성이 로컬 환경에 정상적으로 설치되지 않은 상태에서 작업 진행

**해결 과정**

- CI 로그를 통해 실패한 코드를 분석
- 누락된 의존성 및 플러그인 설치
- 최신 develop 브랜치와 동기화 후 rebase를 통해 PR업데이트

**결과**

- CI 빌드 정상 통과
- Husky + CI 기반 코드 검증 흐름 경험

## retrospect

> 개인 개발과 팀 개발의 차이점 인식

기존 1인 개발 프로젝트에서는 혼자서 개발 환경을 구축하고,

발생한 문제 역시 개인의 판단으로 빠르게 수정하는 방식에 익숙했다.

이번 사이드 프로젝트에서는 브랜치 전략, PR 기반 코드 리뷰, CI 검증 과정, 공통 컴포넌트 및 디자인 토큰 관리 등을 경험하며,

좋은 코드란 단순히 동작하는 코드가 아니라 팀 단위의 의사결정과 규칙 속에서 유지될 수 있는 코드라는 점을 체감했다.

특히 코드 리뷰 과정에서 구조 개선에 대한 피드백을 받으며,

기능 구현 이후에도 다른 개발자가 이해하고 확장할 수 있는 구조를 고려하는 습관이 필요함을 느꼈다.

이후에는 구현 단계부터 재사용성과 협업 관점을 함께 고려하며 개발하려 노력하고 있다.

> 팀 프로젝트에 임하는 마음가짐

프로젝트에는 현업에 재직 중인 구성원들도 참여하고 있었으며,

주 1회 전체 회의와 파트별 회의를 포함해

주 2회 이상의 정기적인 협업 과정이 꾸준히 유지되고 있었다.

각자의 본업이 있음에도 지속적으로 참여하며 책임감 있게 프로젝트를 이어가는 모습을 보며,

사이드 프로젝트 역시 실제 서비스 개발과 동일한 태도로 임해야 한다는 인식을 갖게 되었다.

이후 회의 참여와 작업 일정 관리에 더욱 책임감을 가지고 임하게 되었고,

단순 참여가 아닌 팀의 진행 흐름에 기여하려는 자세로 개발에 임하고 있다.
