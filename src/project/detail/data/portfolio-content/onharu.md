## description

> 프로젝트 소개

- 나눔카드를 드러내지 않고도, 아이들이 자연스럽게 동네의 나눔 가게를 이용할 수 있도록 연결하는 서비스

> 주요 기능

- 아동 / 가게 유형별 회원 시스템 : 사용자 역할에 맞는 회원가입·로그인 제공
- 위치 기반 나눔가게 탐색 : 내 주변에서 이용 가능한 나눔 가게를 지도 기반으로 확인
- 비노출 예약 시스템 : 나눔카드를 드러내지 않고 일반 예약처럼 서비스 이용
- 실시간 채팅 : 예약 전·후 가게와 직접 소통 가능

> 기여 내용

- 서비스 문제 정의부터 기획·UI/UX 설계 전반 주도
- 재사용성을 고려한 공통 컴포넌트 시스템 설계 및 구현
  (Button, Modal, Toast, Card, Pagination)
- 메인·리스트·상세 페이지 사용자 흐름 기반 구현
- 위치 기반 주변 가게 탐색 및 비노출 예약 시스템 구현
- 모바일 환경을 고려한 Bottom Sheet 인터랙션 구현

## skills

> Next.js (App Router)

- App Router 기반으로 URL 중심 상태 관리 설계에 적합하며 SSR 전략으로 초기 로딩 최적화

> Typescript

- API 응답 타입 명확화로 런타임 오류 최소화
- 협업 안정성 확보를 위해 도입

> Tanstack Query

- 서버 상태 캐싱 및 자동 리패칭 관리, 불필요한 중복 요청 방지.

> Tailwind CSS

- 빠른 프로토타이핑 및 팀원 간 일관된 스타일 유지에 적합

## trouble

> URL 기반 단일 상태 관리 훅 도입으로 상태 동기화 문제 해결

**문제**

- 카테고리, 정렬, 페이지, 위치, 검색 기능이 추가되면서 상태가 state와 URL에 동시 존재
- 새로고침 시 필터 상태 초기화
- 뒤로가기/URL 공유 불가
- 필터 로직이 Nearby 컴포넌트 내부에 흩어져 있어 코드 복잡도 증가

**해결**

- URL을 단일 상태 소스로 정의, 각 핸들러는 URL의 파라미터를 변경시키고 파라미터가 변경되며 api를 호출하는 단일 로직으로 변경
- useStoreFilter 커스텀 훅을 만들어 필터/정렬/검색/위치 로직 캡슐화
- TanStack Query와 URL을 연동하여 URL 변경 시 자동 데이터 refetch

**결과**

- 새로고침/공유/뒤로가기 시 상태 유지 가능
- 필터 로직 중복 제거 및 재사용 가능성 상승
- 서버 상태(TanStack Query)와 URL 상태의 명확한 역할 분리

> 데스크톱/모바일 디자인 분리로 인해 컴포넌트가 무거워짐

**문제**

- 모바일/데스크톱 대응을 위한 조건부 렌더링과 여러 상태 관리 로직이 한 컴포넌트에 집중되면서
  page.tsx가 200줄 이상으로 비대해져 가독성과 유지보수성이 저하됨.

**분석**

- Import 문이 30줄 이상
- 상태 관리 로직(useState, useEffect)과 UI 렌더링이 혼재
- 모바일/데스크톱 UI 코드 중복 (StoreCardList 렌더링 로직 2번 반복)
- 코드 가독성 저하로 신규 기능 추가 시 어디를 수정해야 할지 파악이 어려움

**해결**

- Container/Presentational Pattern을 적용하여 로직과 뷰를 분리
- Container Component는 상태관리와 데이터 패칭을 담당
- Presentational Component는 UI 렌더링과 props 전달만을 담당

**결과**

- 가독성 향상: 200+줄 → 70+줄 (약 60% 이상 감소)
- 유지보수 개선 : 로직 수정 시 page.tsx만, UI 수정 시 View 컴포넌트만 수정
- 코드 재사용성 : StoreCardList를 Desktop/Mobile 양쪽에서 재사용

## retrospect

> 역할과 책임

온하루에서 나는 기획, 웹디자인, 프론트엔드를 담당했다.

특히 로그인하지 않아도 가장 먼저 노출되는 UI 영역을 맡았기 때문에, 사용자 경험의 첫 인상을 만든다는 책임감을 가지고 작업했다.

기획과 디자인을 선행하며 전체 화면 흐름을 정리했고, 문서화 작업을 통해 팀원들이 빠르게 개발에 진입할 수 있도록 기반을 만들었다.

> 일정과 현실적인 제약

가장 어려웠던 부분은 일정 조율이었다.

서버 담당이 재직 중이었기 때문에 실무처럼 촉박한 일정 속에서 개발이 진행되었다.

백엔드 일정이 맞지 않아 목데이터로 먼저 퍼블리싱을 진행했지만, 이후 전달받은 API 명세가 기존 목데이터와 달라 구조를 크게 수정해야 했다.

이 경험을 통해 “가정 기반 설계는 언제든 수정 비용이 발생한다”는 점을 체감했다.

또한 팀 인원이 적어 각자 맡은 영역 개발에 집중하다 보니, 코드 리뷰가 즉각적으로 이루어지기 어려웠다.

그 결과, merge 시점에서 예상치 못한 충돌이나 구조 문제를 발견하는 경우도 있었다.

> 내가 팀에 기여한 부분

가장 크게 기여했다고 생각하는 부분은 다음과 같다.

- 기획 및 디자인 문서화로 초기 개발 방향성 제시

- 공통 컴포넌트(Card, Pagination, Modal, Button) 설계 및 구현

- URL 기반 API 요청을 위한 커스텀 훅 제작

- 재사용성을 고려한 구조 설계

특히 공통으로 사용될 수 있는 컴포넌트를 먼저 제작하고, 팀원들이 사용하는 과정에서 불편함은 없는지 지속적으로 확인하려 노력했다.

팀원들로부터 "초기 아웃풋 덕분에 개발 속도가 빨라졌다"는 피드백을 받았던 점은 개인적으로도 의미 있는 경험이었다.

> 협업을 통해 배운 점

개인 프로젝트와 달리, 팀 프로젝트에서는 “내 코드”가 아니라 “우리 코드”라는 점을 배웠다.

한 번은 공통 컴포넌트를 수정하면서 충분히 공유했다고 생각했지만,

다른 팀원이 해당 컴포넌트를 다른 방식으로 사용하고 있었다는 사실을 merge 과정에서 알게 되었다.

이 경험을 통해 공통 영역 수정 시에는 반드시 사전 공유와 동의가 필요하다는 점을 깨달았다.

또한 프로젝트 초반에 명확한 리더십이 부재하면서 방향성이 흔들리는 경험도 했다.

이후 자료와 근거를 정리해 제안하는 방식을 택했고,

단순 의견이 아니라 근거 기반 제안이 설득력을 만든다는 것을 체감했다.

한편, 다른 팀 프로젝트에서는 4명의 팀원이 비교적 여유 있는 일정 속에서 꼼꼼한 코드리뷰 문화를 경험한 적이 있다.

그 과정에서 개발 경력이 많은 팀원의 피드백을 통해 새로운 라이브러리와 설계 방식, 더 나은 코드 구조를 배울 수 있었다.

이 경험과 비교해보며, 온하루 프로젝트에서는 일정에 쫓기다 보니 코드리뷰가 사후 점검에 가까웠다는 점이 아쉬움으로 남았다.

다시 진행한다면, 기능 개발만큼 코드리뷰에도 시간을 확보해 팀 전체 코드의 품질을 끌어올리고 싶다.

> 아쉬움과 개선 방향

인력과 시간의 제약은 현실적인 한계였다.

하지만 이는 실무에서도 반복되는 문제이기에, 오히려 실제 환경과 유사한 경험이었다고 생각한다.

다시 진행한다면 초기 단계에서 API 스펙 확정 여부를 명확히 하고,

공통 컴포넌트 수정 프로세스를 팀 규칙으로 정했을 것이다.
